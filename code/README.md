# Support Triage Agent

## What the Agent Does

This agent automatically triages customer support tickets for three companies — **HackerRank**, **Claude (Anthropic)**, and **Visa** — using a Retrieval-Augmented Generation (RAG) pipeline. For each incoming ticket it first applies fast, deterministic rule-based gates to catch invalid inputs (gibberish, competitor references, prompt-injection attempts) and high-risk tickets that require a human (fraud, legal threats, account compromise, billing disputes). Tickets that pass both gates are scored against a local corpus of ~774 support articles; the top-3 most relevant articles are injected as grounding context into a prompt sent to the **Fireworks DeepSeek** model, which produces a structured JSON response containing a customer-facing reply, a routing status, a product area label, a request type classification, and a one-sentence justification citing which articles it relied on.

## Installation

Requires Python 3.9+.

```bash
pip install -r code/requirements.txt
```

## Setting the API Key

The agent uses the Fireworks AI inference endpoint (OpenAI-compatible). Set your Fireworks API key as the `OPENAI_API_KEY` environment variable:

```bash
export OPENAI_API_KEY=your-key-here
```

Or copy `.env.example` to `.env` and fill it in — load it before running:

```bash
cp .env.example .env
# edit .env, then:
source .env
```

> The `.env` file is gitignored and must never be committed.

---

## Running

From the **repo root**:

```bash
python code/main.py
```

The runner will:
1. Load all ~774 corpus articles once into memory.
2. Read every row from `support_tickets/support_tickets.csv`.
3. Write `support_tickets/output.csv` when finished.

For a quick smoke-test of the agent on a single hardcoded ticket:

```bash
python code/agent.py
```

For a quick smoke-test of the retriever:

```bash
python code/retriever.py "how do I cancel my Claude Pro subscription"
```

## Output Format

`support_tickets/output.csv` contains one row per input ticket with these columns:

| Column | Type | Description |
|---|---|---|
| `issue` | string | Original ticket body (copied from input) |
| `subject` | string | Original ticket subject (copied from input) |
| `company` | string | Company the ticket belongs to (copied from input) |
| `status` | `replied` \| `escalated` | `replied` — agent produced an answer; `escalated` — routed to human |
| `product_area` | snake_case string | Most relevant product area (e.g. `billing`, `screen`, `travel_support`) |
| `response` | string | Customer-facing reply composed from corpus context |
| `justification` | string | 1–2 sentences explaining the routing decision and which articles were used |
| `request_type` | `product_issue` \| `feature_request` \| `bug` \| `invalid` | Classification of the ticket type |

## Design Decisions

### Why RAG instead of fine-tuning?

Fine-tuning a model on support content requires labelled training data, GPU infrastructure, and periodic retraining whenever the knowledge base changes. RAG separates the *retrieval* of up-to-date facts from *generation*, so adding or editing a support article is as simple as dropping a `.md` file into `data/`. This suits a hackathon timeline and mirrors how production support-bot systems are actually built.

### Why keyword retrieval instead of embeddings?

Dense vector search (e.g. FAISS + a sentence-transformer) would give better semantic recall but requires a sizeable embedding model or an external API call per ticket, adding latency, cost, and an extra dependency. The corpus is structured with clear titles, breadcrumbs, and domain-specific jargon (`chargeback`, `rate limit`, `workbench`) that keyword overlap already captures well. The scorer applies weighted boosts for title matches, breadcrumb matches, and company/product-area hints to compensate for the lack of semantic similarity. For a larger or more ambiguous corpus, switching to a hybrid BM25 + embedding approach would be the natural next step.

### Why rule-based escalation instead of LLM-based?

High-stakes signals — fraud, legal threats, account compromise — must be caught *before* an LLM call, not left to the model's discretion. A regex gate is:
- **Deterministic**: identical input always produces the same routing decision.
- **Auditable**: every escalation pattern can be reviewed and adjusted without touching the model.
- **Fast and cheap**: no API call is needed for tickets that match, protecting against latency spikes.
- **Robust to prompt injection**: a jailbroken model might be tricked into not escalating; a regex never will be.

LLM-based classification is used only for the nuanced parts (product area, request type, response wording) where rigid rules would break on natural language variation.

### Where the agent may fail and how it could be improved

| Failure mode | Root cause | Improvement |
|---|---|---|
| Missing context → escalation instead of answer | Keyword retriever misses semantically relevant articles with different vocabulary | Hybrid BM25 + embedding retrieval; query expansion |
| Wrong product area label | Model generalises from nearby context | Add a fixed taxonomy and use constrained decoding or a post-processing lookup table |
| Escalation gate over-triggers | Broad regex patterns match benign text (e.g. "dispute" in "I'd like to dispute an invoice amount") | Tighten patterns with negative lookaheads; add a lightweight intent classifier as a second gate |
| Competitor filter blocks legitimate questions | Regex is keyword-based and can't infer intent | Replace with a small classifier or add an allow-list for context where the competitor name is incidental |
| No retry on transient API errors | Single attempt per ticket | Add exponential-backoff retry with a maximum of 3 attempts before falling back to escalation |
| Long tickets exceed model context | No truncation of very long issue bodies | Truncate the ticket to the first N tokens; summarise if needed before retrieval |
